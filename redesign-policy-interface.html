Redesign policy interface

Apr-Oct 2021

Policy interface was one of the business focuses in 2021. We had plans to integrate new features into this interface but before it, we needed to research user needs and redesign the policy interface to be more flexible to new changes.

User research ✷ Usability testing ✷ Prototyping ✷ UI/UX design

Before

Policy interface had not changed for several years because the product was focused on the agents' works with quotes and submissions. In 2021 plans had been changed, the business put the policy to the product roadmap and set several goals on adding big features to it. But the interface had already been clunky and outdated, and did not respond to user needs.

When users opened policy firstly they saw questionnaire and table. The main info was hidden on the last tab

After

Even when thinking about business performance, it is important to keep the user-centered approach in mind. Therefore, we studied the problems of users on this interface and updated it in accordance with users’ goals, paving the way for the further addition of features and the growth of business metrics.

User research

We started by researching the data of the interface's usage. We checked the quantitative data to find out the attendance of the tabs and the popular actions on them. After that, we started collecting quality data:

— interviewed stakeholders and sales team
— conducted a competitive analysis
— reviewed support tickets

Based on this data, we made an analog of CJM for features. It described what goals the users had and what information and actions are needed to achieve these goals.

As a result, we found out that the old interface had redundant navigation. The main info was placed on the different tabs. To put the info together user needed to move between tabs.

Solution design

The main idea I followed was creating the main summary page for policy. Users can get back to the policy after several months after working on it to check the info or upload endorsements docs. All main info needed to be at a glance. Also, I needed to keep in mind that this page will be the starting point for adding new features.

Prototype testing

I built a Figma prototype and with help from the sales team tested it with 5 users to validate the solution. We asked our users to complete 4 tasks and counted the number of completed tasks and time for each task.

As a result, we approved that one page with all main info helps users to easier find policy documents and main data. Also we received useful comments to update the prototype.

Iterative approach

We started to move iteratively by defining key metrics and checking them on each stage and making changes if needed.

1st iteration. Added header with insurance line and moved there main actions, the form was transferred to new fields from the design system

2nd iteration. Renamed Options tab to Policy and placed it on the first place. Deleted Policy Docs tab and moved documents table to the new tab and redesigned it. Made a card with the main policy info, moved editing this info to dialog

3rd iteration. Added detailed information about policy coverages

Outcomes

For each iteration, we specified metrics to track how changes impacted users’ behavior and to change the plan if something went unexpected. Except for the usual metrics such as bounce rate, time on page, we built a funnel to see how switching tabs had affected userflow.
